Three Architectural Approaches

Enhanced CVE display + SIGMA engine revision — pick the approach

A

Surgical Enhancement

Change the minimum to unblock issue #25.

  • Extend Finding with an evidence: List<Evidence> payload
  • Evaluator emits N atomic findings for CVE rule
  • Add OSV merge to CveDatabase, persist to Room
  • Campaign tags on rule YAML
  • Bottom sheet UI for CVE list
  • FindingMapper stays but becomes evidence-aware

Pros

  • Smallest diff, fastest delivery
  • Low regression risk

Cons

  • FindingMapper stays hard-coded
  • Generic rendering not addressed
  • Technical debt accumulates
B

Engine Rethink (Recommended)

Redesign the finding→UI pipeline. Eliminate FindingMapper. Deliver CVE feature on the new architecture.

  • Findings become self-describing: title, severity, description, evidence, remediation, display hints — all from rule YAML
  • Generic renderers with display primitives (severity list, expandable section, tag group, key-value evidence)
  • Evaluator supports 1:N emission (one rule → N atomic findings)
  • FindingMapper eliminated — findings render directly
  • CVE feature built on the new pipeline: Room persistence, OSV merge, campaign tags, bottom sheet
  • Existing 20 rules migrated to carry display metadata

Pros

  • Eliminates hard-coded translation layer
  • Future rules render without Kotlin changes
  • Aligns with public repo vision
  • CVE feature validates the new architecture

Cons

  • Larger scope — touches engine, all rules, UI, data layer
  • Needs careful migration of 20 existing rules
  • Moderate regression risk
C

Full Platform Overhaul

Everything in B, plus future-facing platform capabilities.

  • Everything from approach B
  • Rule schema versioning (v1 → v2 migration support)
  • Rule dependencies and chaining (rule A's output feeds rule B)
  • SARIF/OCSF export format for findings
  • Finding lifecycle (new → acknowledged → resolved → suppressed)
  • Per-finding dismiss/snooze in UI

Pros

  • Most complete long-term architecture
  • Export-ready data model
  • Professional finding management

Cons

  • Massive scope — weeks of work
  • YAGNI risk: building for hypothetical needs
  • Delays the CVE feature significantly
  • Rule chaining adds complexity with no current consumer
